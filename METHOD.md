## План работы по обзору

### 0 Структура и оформление

Материалы и методы по обзору (где под материалами обзора понимаем вспомогательные сведения, требуемые систематической методологией, собранные в процессе проведения обзора, вроде блок-схемы PRISMA и библиометрической структуры рассматриваемых подборок исследований) отражаем для удобства в соответствии с пунктами [PRISMA-ScR](http://prisma-statement.org/Extensions/ScopingReviews) из раздела «Methods» для методов обзора и из раздела «Results» для материалов обзора. Отсылки к пунктам удалим (для лучшей читаемости) в самый последний момент, чтобы у нас была также и версия с помеченными пунктами, что может быть удобно при написании потом статьи, если решим как-то комплаентность с PRISMA-ScR там отразить.

Все аннотации постараемся согласовывать с PRISMA-Abstracts.

Отрабатывать методы будем в порядке следования этапов обзора (оригинальная схема):

1. Планирование и регистрация.
2. Поиски.
3. Скрининг библиографических записей (1-й этап скрининга).
4. Скрининг полных текстов (2-й этап скрининга).
5. Извлечение данных (картирование).
6. Качественный синтез данных и оценка их качества.
7. Количественный синтез данных.
8. Представление и публикация результатов.

Подключим к работе и других участников.

[SFMHSUSH SSB HSSS v2](https://github.com/p1m-ortho/xr-sfmhsush-ssb-hsss-v2) планируем, где необходимо (см. ниже), использовать в качестве тренировочного материала. Этот обзор вроде бы совместим по критериями включения, хотя этот вопрос нуждается в уточнении. Также этот обзор нуждается в завершении скрининга.

Таким образом, для использования SFMHSUSH SSB HSSS v2 здесь необходимо (а) убедиться в совместимости обзоров и документировать эти соображения и (б) завершить в нем перекрестный скрининг, документировав результаты.

Дело в том, что, как мы представляем, регистрация призвана обеспечить возможность (а) раннего обнаружения препринта протокола научным сообществом, особенно исследователями, планирующими сходные работы, чтобы исключить дублирование усилий, и (б) сравнить итоговую работу с изначальной версией протокола, не доступной после регистрации изменению со стороны авторов. Реализация пункта (а) на OSF — сомнительна, поскольку не имеем свидетельств ее популярности в качестве регистра обзоров литературы, однако необходимо попытаться такие свидетельства получить.

### 3 Скрининг библиографических записей (1-й этап скрининга)

Перекрестный, чтобы рассчитать показатели внешнего согласия. Планируем подключить нескольких участников, затем, возможно, случайным образом (RANDOM.ORG или другой TRNG) выборки некоторого объема всем раздать в две или более раздачи (в пределах раздачи каждому участнику — без повторений, в каждой последующей — с повторениями). Тогда показатели внутреннего согласия тоже можно будет рассчитать. Посмотрим на эту тему во всей теории выше, плюс при необходимости точечно поднимая или выискивая еще точечно статьи.

Идею каскадного исключения по шагам оставляем в силе, конкретный список шагов будем уточнять по мере прояснения поисковой стратегии.

Скрининг по записям будем проводить в Rayyan в слепом режиме. Затем списки отдельно включенных и исключенных (возможно, в отдельности по каждому из шагов) для каждого участника экспортируем в BibTeX и выгрузим в репозиторий. Затем снимем слепой режим и проведем согласование. Результаты в том же формате выгрузим в репозиторий. Отразим в блок-схеме PRISMA.

### 5 Извлечение данных (картирование)

Карта в нашем случае представлена блоками обзора. То есть заполнение будет представлено вписыванием текста в соответствующее блоку место обзора.

По каждому из блоков пройдем (еще до публикации протокола) и как можно точнее сформулируем, какая информация для него подходит. Затем соберем участников и обсудим, уточним пункты.

Картирование каждый участник будет выполнять в своей ветке.

Принципиально видим два подхода к картированию: (а) для данного блока проходить по каждой включенной статье или (б) для данной включенной статьи проходить по каждому блоку.  Вероятно, предоставим участникам возможность картировать свою подвыборку в удобном им варианте в пределах каждой единицы картирования (соответственно один блок или одна статья), при этом будем просить делать новый коммит после внесения каждой новой единицы и в его заголовке указывать, какой подход применяли. Затем планируем учитывать влияние варианта картирования при расчете показателей согласия.

Отпилотируем картирование еще до публикации протокола. Для этого планируем подключить нескольких участников, затем, возможно, случайным образом (RANDOM.ORG или другой TRNG) выборки некоторого объема из включенных в [SFMHSUSH SSB HSSS v2](https://github.com/p1m-ortho/xr-sfmhsush-ssb-hsss-v2) всем раздать в две или более раздачи (в пределах раздачи каждому участнику — без повторений, в каждой последующей — с повторениями). Тогда показатели внутреннего согласия тоже можно будет рассчитать. Посмотрим на эту тему во всей теории выше, плюс при необходимости точечно поднимая или выискивая еще точечно статьи.

При неудовлетворительных показателях согласия соберем обсуждение еще раз и создадим финальный вариант формы, который вновь отпилотируем по тому же пути.

Собственно картирование тоже планируем производить, раздав каждому из участников некоторое количество случайных фрагментов в две или более раздач. Результатом будет по варианту обзора в ветке каждого из участников.

### 6 Качественный синтез данных и оценка их качества

Качественный синтез в пределах блоков будет происходить по мере картирования в необходимой, с точки зрения картирующего, мере. Формальную оценку качества проводить не планируем.

Синтез между блоками: каждый из участников синтезирует (технически: смерджит в свою ветвь) финальный вариант обзора и напишет резюме по обзору, затем общим собранием будет согласован единый общий вариант (технически: будет смерджен из ветвей всех участников в «мастер»).
